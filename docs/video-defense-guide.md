## 二、答辩主线叙事（建议开场 2–3 分钟）

### 2.1 一句话定位

> 本系统是**校园湖海试验场数字孪生全景监控与数据管理系统**，面向船舶与海洋工程试验场，实现从「学生预约试验」到「AI 生成分析报告」的**完整业务闭环**，并体现前后端分离、分层架构、主从表业务、WebSocket 实时推送、国产达梦数据库与 DeepSeek AI 集成等课程要求。

### 2.2 业务主线（答辩核心故事）

用下面这条链路贯穿整段录屏，每完成一步可口头点一下「当前处于哪一阶段」：

```text
📝 预约草稿
  → 📤 提交预约（资源冲突校验）
  → 👨‍🏫 教师审核
  → 👔 主任审批
  → 📋 自动生成试验任务
  → 🔧 任务准备
  → ▶️ 启动试验
  → 📡 数字孪生实时监控与告警
  → 📦 试验归档与轨迹回放
  → 🤖 AI 智能分析报告
```

### 2.3 技术主线（穿插讲解，不必单独录长段）

| 层次 | 技术 | 答辩可强调的点 |
|------|------|----------------|
| 前端 | Vue 3 + TS + Element Plus + ECharts + Three.js | 组件化 SPA、驾驶舱图表、3D 孪生场景 |
| 后端 | FastAPI + SQLAlchemy + Pydantic | REST + WebSocket，Swagger 可现场打开 |
| 架构 | API → Service → Repository → Model | **复杂业务不写在 Controller**，Service 层承载审批流、冲突校验、状态机 |
| 数据库 | SQLite（开发）/ 达梦 DM8（答辩） | 主从表、外键、逻辑删除、审计字段 |
| 实时 | WebSocket（默认）/ MQTT（可选） | 监控帧推送、设备指令回执 |
| AI | DeepSeek API（后端调用） | Key 不暴露前端；Mock 与真实 API 可切换 |

### 2.4 设计亮点（挑 3–4 个重点讲）

1. **主从表 + 两级审批**：`EXP_RESERVATION`（预约头表）+ `EXP_RESERVATION_RESOURCE`（资源明细从表）；教师 → 主任两级审批后自动生成试验任务。
2. **资源冲突校验**：提交时预校验 + 主任审批前终校验；非简单 CRUD，体现业务规则。
3. **试验任务状态机**：`待准备 → 就绪 → 运行中 → 已完成 → 已归档`，状态流转有约束。
4. **智能中控台**：视频感知 + CV 识别 + Three.js 孪生联动 + 实时曲线 + 设备控制 + 系统健康条。
5. **试验全生命周期数据**：传感器、轨迹、告警、文件、AI 报告统一归档，支持时间轴回放与导出。

## 四、各模块「介绍什么」速查表

## 六、答辩常见问题与应答要点

| 问题 | 应答要点 |
|------|----------|
| 主从表在哪体现？ | `EXP_RESERVATION` + `EXP_RESERVATION_RESOURCE`；预约详情页上主下从；一条预约多条资源明细 |
| 和简单 CRUD 有何不同？ | 两级审批流、资源冲突校验、状态机、审批后自动生成任务 |
| 实时数据从哪来？ | 默认后端 WebSocket 模拟推送；可选 MQTT 接入真实/模拟设备 |
| AI Key 放哪？ | 仅 `backend/.env`，`ai_service.py` 调用，前端不接触 |
| 达梦怎么切换？ | `DATABASE_URL` 指向 DM8，执行 `init_db.sql` + `seed_db`；工作台显示库类型 |
| 分层架构怎么体现？ | 指 `api/`、`services/`、`repositories/`、`models/`；业务在 Service |
| 数字孪生是否真实硬件？ | 课程阶段允许模拟；架构预留 MQTT、Edge Agent、视频流接口 |
