## 2. 项目概述

### 2.1 系统名称

校园湖海试验场数字孪生全景监控与数据管理系统。

### 2.2 建设背景

高校船舶与海洋工程湖海试验场承担教学实验、科研验证、模型船测试和新技术验证等任务，涉及拖曳水池、风浪流水槽、深浅湖、模型船、传感器、摄像头、造波设备、拖车设备等多类资源。传统管理方式主要依赖人工预约、纸质记录和现场值守，存在资源调度效率低、试验过程感知不足、试验数据分散、异常预警能力弱、试验结果难以复盘等问题。

本系统拟通过数字孪生、物联网数据采集、全景监控、实时告警和 AI 智能分析技术，构建面向校园湖海试验场的综合管理平台，实现试验资源预约审批、试验过程可视化、模型船轨迹监控、传感器数据采集、试验数据归档和智能分析，提升试验教学和科研管理效率。

### 2.3 总体目标

系统建设目标如下：

1. 实现试验任务从预约、审批、准备、执行、分析到归档的全流程管理。
2. 实现实验场地、模型船、传感器、摄像头、拖车、造波设备等资源的统一管理。
3. 实现基于数字孪生场景的试验过程可视化监控。
4. 实现模型船位置、速度、姿态、电量、传感器数据等信息的实时采集与展示。
5. 实现模型船越界、设备离线、电量过低、数据突变等异常告警。
6. 实现试验任务、轨迹数据、告警数据、文件资料和 AI 报告的统一归档。
7. 接入 DeepSeek API，实现试验摘要、异常解释、风险提示和复盘建议生成。
8. 使用达梦 DM8 作为核心业务数据库，体现国产数据库应用。

## 3. 设计原则

### 3.1 前后端分离

系统前端采用 Vue 3 构建界面，后端采用 FastAPI 提供 RESTful API 与 WebSocket 服务。前后端通过 JSON 数据交互，便于并行开发、独立测试和后续维护。

### 3.2 分层架构

后端采用接口层、业务服务层、数据访问层、数据模型层和数据传输层分离的结构。复杂业务逻辑不得直接写在 API 路由中。

### 3.3 可演示优先

本项目服务于课程设计展示，因此优先保证业务闭环完整、主从表清晰、角色权限可演示、实时监控可运行、AI 接口可调用。

### 3.4 轻量可扩展

课程设计阶段允许使用模拟数据、简化数字孪生模型和本地文件存储，但接口和数据模型应保留未来接入真实设备、视频流和消息中间件的扩展空间。

### 4.3 主业务流程

课程设计 v1 统一采用固定两级审批流程，不再区分“部分试验是否跳过主任审批”。

```text
学生/研究员创建预约草稿
        ↓
学生/研究员提交预约
        ↓
指导教师审核
        ↓
试验场主任审批
        ↓
系统生成试验任务
        ↓
任务准备
        ↓
现场试验执行
        ↓
数字孪生监控 + 数据采集 + 告警
        ↓
试验完成
        ↓
归档
        ↓
DeepSeek 生成分析报告
```

## 7. 核心业务规则

## 7.1 预约状态机

### 7.1.1 预约状态枚举

课程设计 v1 统一使用以下状态：

| 状态值 | 含义 |
| --- | --- |
| `DRAFT` | 草稿，允许编辑资源明细 |
| `PENDING_TEACHER` | 已提交，待教师审核 |
| `PENDING_DIRECTOR` | 教师已通过，待主任审批 |
| `APPROVED` | 主任已批准，已生成试验任务 |
| `REJECTED` | 被驳回 |
| `CANCELLED` | 已取消 |
| `COMPLETED` | 对应试验任务已完成 |
| `ARCHIVED` | 已归档 |

### 7.1.2 预约流转规则

```text
DRAFT
  └─提交→ PENDING_TEACHER

PENDING_TEACHER
  ├─教师通过→ PENDING_DIRECTOR
  ├─教师驳回→ REJECTED
  └─申请人取消→ CANCELLED

PENDING_DIRECTOR
  ├─主任通过→ APPROVED
  ├─主任驳回→ REJECTED
  └─申请人取消→ CANCELLED

APPROVED
  ├─试验完成→ COMPLETED
  └─归档完成→ ARCHIVED
```

### 7.1.3 审批规则

1. 本期统一采用“教师审核 + 主任审批”两级审批。
2. 驳回必须填写原因。
3. 已取消、已完成、已归档的预约不得再次审批。
4. 审批操作必须记录操作人、操作时间和审批意见。

## 7.2 资源冲突与占用规则

### 7.2.1 资源状态

| 状态值 | 含义 |
| --- | --- |
| `AVAILABLE` | 可用 |
| `RESERVED` | 在未来时间段已被预约占用 |
| `IN_USE` | 当前试验执行中 |
| `MAINTENANCE` | 维护中 |
| `FAULT` | 故障 |
| `DISABLED` | 停用 |

### 7.2.2 资源预约校验

为避免多个待审批预约相互冲突，本系统采用“提交时预校验 + 审批前终校验”策略。

1. 创建草稿时不锁定资源。
2. 提交预约时，校验资源是否与以下状态的预约冲突：
   `PENDING_TEACHER`、`PENDING_DIRECTOR`、`APPROVED`。
3. 教师审核时只处理业务可行性，不改变资源锁定规则。
4. 主任审批通过前，必须在同一事务中再次校验资源冲突。
5. 主任审批通过后，预约转为 `APPROVED`，并生成试验任务。
6. 试验开始执行时，相关资源状态由 `RESERVED` 变为 `IN_USE`。
7. 预约取消或驳回后，应释放其对未来时段的预约占用。
8. `MAINTENANCE`、`FAULT`、`DISABLED` 状态资源不能被预约。

### 7.2.3 主从表事务规则

1. 创建预约时，主表和资源明细表必须在同一事务中保存。
2. 修改草稿预约时，主表与明细更新必须保持事务一致。
3. 主任审批通过并生成试验任务时，预约状态更新与任务创建也必须在同一事务中完成。

## 7.3 试验任务状态机

### 7.3.1 任务状态枚举

| 状态值 | 含义 |
| --- | --- |
| `PENDING_PREPARE` | 审批完成，待准备 |
| `READY` | 资源和设备已准备完毕 |
| `RUNNING` | 试验执行中 |
| `COMPLETED` | 试验完成 |
| `ARCHIVED` | 数据与资料已归档 |
| `CANCELLED` | 已取消 |

### 7.3.2 任务流转

```text
PENDING_PREPARE → READY → RUNNING → COMPLETED → ARCHIVED
```

说明：

1. `APPROVED` 预约创建 `PENDING_PREPARE` 任务。
2. 进入 `RUNNING` 后允许实时监控、传感器入库和告警生成。
3. 仅 `COMPLETED` 或 `ARCHIVED` 的任务允许生成正式 AI 报告。

## 7.4 删除与审计规则

1. 预约、任务、告警、文件、AI 报告等重要业务数据默认采用逻辑删除。
2. 资源若已被历史预约引用，不建议物理删除，应改为 `DISABLED`。
3. 关键审批、任务状态流转、资源状态变更必须记录操作日志。

## 8.2 试验预约与审批模块

### 模块目标

实现试验预约申请、资源明细维护、两级审批和试验任务生成，是课程设计的核心业务模块。

### 主从表设计

主表：`EXP_RESERVATION`

从表：`EXP_RESERVATION_RESOURCE`

关系：

```text
EXP_RESERVATION 1 —— N EXP_RESERVATION_RESOURCE
```

### 主要业务规则

1. 预约开始时间必须早于结束时间。
2. 同一资源在同一时间段内不能被重复预约。
3. 预约草稿允许多次编辑资源明细。
4. 提交后进入教师审核。
5. 教师审核通过后进入主任审批。
6. 主任审批通过后生成试验任务。
7. 驳回必须填写原因。

### 主要接口

| 接口 | 方法 | 说明 |
| --- | --- | --- |
| `/api/reservations` | GET | 查询预约列表 |
| `/api/reservations/{id}` | GET | 查询预约详情 |
| `/api/reservations` | POST | 新增预约草稿 |
| `/api/reservations/{id}` | PUT | 修改草稿预约 |
| `/api/reservations/{id}/submit` | POST | 提交预约 |
| `/api/reservations/{id}/teacher-review` | POST | 教师审核 |
| `/api/reservations/{id}/director-approve` | POST | 主任审批 |
| `/api/reservations/{id}/cancel` | POST | 取消预约 |

## 8.5 实时数据采集与告警模块

### 模块目标

实现试验过程数据的实时采集、存储、推送和异常告警。

### 数据类型

1. 模型船 X / Y 坐标。
2. 速度。
3. 航向角。
4. 横摇角、纵摇角。
5. 电池电量。

5. 设备离线。

## 8.7 DeepSeek 智能分析模块

### 模块目标

接入 DeepSeek API，实现试验任务的自然语言分析，辅助教师和学生理解试验结果。

### 功能设计

1. 生成试验概况。
2. 生成关键数据摘要。
3. 分析异常原因。
4. 生成风险提示。
5. 给出改进建议。
6. 保存 AI 分析报告。

### 调用规则

1. DeepSeek API 只允许后端调用。
2. 前端不得保存或暴露 API Key。
3. 生成 AI 报告时优先传入结构化摘要，而不是大量原始数据。

## 10. 接口总体设计

### 10.1 接口设计原则

1. 接口路径统一以 `/api` 开头。
2. 普通 HTTP 接口统一返回 `{ code, message, data }`。
3. 需要登录的接口必须携带 Token。
4. 关键业务接口需要进行角色校验和状态校验。
5. WebSocket 独立使用 `/ws/monitor/{experiment_id}`。

### 10.2 接口分组

| 分组 | 路径前缀 | 说明 |
| --- | --- | --- |
| 登录认证 | `/api/auth` | 登录、退出、当前用户 |
| 用户管理 | `/api/users` | 用户增删改查 |
| 角色权限 | `/api/roles` | 角色管理 |
| 试验预约 | `/api/reservations` | 草稿、提交、审批、取消 |
| 试验资源 | `/api/resources` | 资源设备管理 |
| 试验任务 | `/api/experiments` | 任务准备、启动、完成、归档 |
| 实时监控 | `/api/monitor` | 监控快照与历史查询 |
| 告警管理 | `/api/alarms` | 告警查询和处理 |
| 文件归档 | `/api/files` | 文件上传下载 |
| AI 分析 | `/api/ai` | AI 报告生成与查询 |
| WebSocket | `/ws/monitor/{experiment_id}` | 实时数据推送 |

详细接口定义见 [api-design.md](/D:/CourseDesign/lakesea-digital-twin/docs/api-design.md)。

## 13. 总结

本项目采用 Vue 3 + FastAPI + 达梦 DM8 的前后端分离架构，以“试验预约主表 + 预约资源明细从表”为核心业务设计，以数字孪生监控、WebSocket 实时推送和 DeepSeek 智能分析作为展示亮点。

课程设计 v1 的关键收敛点如下：

1. 审批流程统一为固定两级审批。
2. 资源冲突统一采用“提交预校验 + 审批终校验”。
3. 试验任务补充明确状态机。
4. 审批与状态流转必须保留审计信息。
5. 重要业务数据默认逻辑删除。

按照本版文档实施，可更稳定地支撑后续前后端编码、联调和课程答辩展示。
